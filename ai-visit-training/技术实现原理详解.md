## 1. RAG 检索引擎

### 是什么

不依赖任何向量数据库，纯 Python 实现的知识库检索器，用于为 AI 提供准确的上下文信息。

### 怎么做的

**三维度混合打分：**

```
总分 = 关键词命中(50%) + 短语匹配(35%) + Jaccard相似度(15%)
```

**维度 1 — 关键词命中 (50%)：**

```python
# 用户问题分词后 → 每个 token 在知识库 chunk 中查命中
if token in chunk_keywords:        得分 += 1.0   # 精确命中关键词表
elif token in chunk_keywords_part: 得分 += 0.5   # 部分匹配
elif token in chunk_text:          得分 += 0.3   # 出现在全文
关键词分 = 命中数 / token总数
```

**维度 2 — 短语匹配 (35%)：**

```python
# 2~4 字连续滑动窗口匹配，越长权重越高
for i in range(len(tokens)):
    for win in [2, 3, 4]:
        phrase = ''.join(tokens[i:i+win])
        if phrase in chunk_text:
            短语分 += win  # 4字短语比2字短语权重更高
```

**维度 3 — Jaccard 兜底 (15%)：**

```python
q_chars = set(用户问题字符)      # 例如 {'云','电','脑','价','格'}
c_chars = set(chunk前500字符)
相似度 = 交集长度 / 并集长度
```

### 为什么不用向量检索

- 向量检索需要 embedding 模型（额外 2GB 内存）或 API 调用（每次 +0.5 秒延迟）
- Demo 阶段知识卡内容是结构化的（产品参数、案例数据），关键词匹配精度已够用
- 这个三维度算法是我从零写的——体现了算法设计能力

### 如果评委问"准确率怎么样"

> "定性评估上看，能精准匹配到对应产品的竞品分析和参数卡。定量评估的话，下一步计划用 AI 回答中引用的数据是否与原文一致作为准确率指标。"

## 2. System Prompt 构建

### 是什么

每次 AI 对话前，动态拼装一段约 2500 字的提示词，让 AI 扮演特定阶段、特定难度下的客户。

### 怎么做的

```python
def call_deepseek(messages, scene, difficulty, phase):
    
    # 步骤1：取最近对话内容 → RAG 检索知识库
    recent = messages[-1].content[:200]
    knowledge = rag.search(recent, scene_id=scene, top_k=12)
    
    # 步骤2：加载阶段配置
    pc = phase_configs[phase]  # 取当前阶段的 rules/templates/pain_points
    
    # 步骤3：拼接 System Prompt（7个段落）
    system_prompt = f"""
    你是模拟对练中的真实客户。场景：{scene} | 难度：{difficulty}
    
    ## 你的背景故事（核心人设）
    {pc['pain_points']}          ← 让 AI 有血有肉，不是冷冰冰的质疑机
    
    ## 当前阶段：{pc['name']}
    客户状态：{pc['customer']}    ← 例如"开始敞开心扉但有所保留"
    评分侧重：{pc['focus']}       ← 例如"需求洞察50%+沟通25%"
    
    ## 阶段专属规则（5条）
    {phase_rules}                ← 例如"每轮必须提及竞品名或价格差异"
    
    ## 话术模板（4条）
    {phase_templates}            ← 例如"B公司报价比你们低30%..."
    
    ## 知识库参考
    {knowledge[:2500]}           ← RAG 检索结果注入
    
    ## 输出格式
    <!--COACH 改进建议 -->
    <!--SCORE {"professionalism":70,...} -->
    
    ## 结束规则：轮数到达{rounds}时结束
    """
    
    # 步骤4：调用 DeepSeek
    payload = {
        "model": "deepseek-ai/DeepSeek-V3",
        "messages": [{"role": "system", "content": system_prompt}] + messages,
        "temperature": 0.8,        # 常规模式偏高，让客户更多变
        "max_tokens": 400          # 限制回答长度
    }
```

### 为什么 temperature=0.8？

> "常规模式是训练场景，需要 AI 客户每次的反应不完全一样——这样销售才不会靠背答案过关。反转模式 temperature=0.3，因为需要 AI 给出准确、一致的产品回答。"

## 3. 五阶段对练体系

### 是什么

将客户拜访拆成 5 个递进阶段，每个阶段客户心态不同、评分侧重不同、训练目标不同。

### 怎么做的

```python
phase_configs = {
    'contact': {    # 👋 初次接触
        'name': '初次接触',
        'pain_points': '你被领导催"数字化转型出成果"，今天接了3个推销电话已不耐烦...',
        'rules': [
            '回复简短(1-2句)，不主动展开',
            '可以说"你说说看""大概什么方向"', 
            '禁止直接抛出质疑——你是来了解不是来吵架的'
        ],
        'templates': ['"嗯，你们主要做哪块的？"', '"大概说说吧，我时间不多"'],
        'focus': '表达沟通40%+需求洞察30%'
    },
    'discovery': {  # 🔍 需求深挖
        ...
    }
    # ... 共 5 个阶段
}
```

### 阶段如何生效

- 用户在页面点击阶段选项 → `state.selectedPhase = 'negotiate'`
- 每次对话请求带上 `phase` 参数 → 后端查 `phase_configs[phase]`
- `pain_points` + `rules` + `templates` 全部注入 System Prompt
- AI 客户的行为完全由阶段配置驱动——同一个场景选不同阶段，客户表现截然不同

## 4. 实时评分机制

### 是什么

AI 每轮回复都携带隐藏的评分数据，前端静默解析后实时渲染——用户看到的是自然对话，不知道背后有评分。

### 怎么做的

**后端**：System Prompt 中要求 AI 在每条回复末尾附上：

```
客户口语化回答...

<!--COACH
本轮你没有先共情就直接反驳了客户的价格质疑，建议先说"您的预算考虑非常合理"再切入差异化价值。
-->

<!--SCORE
{"professionalism": 65, "communication": 50, "needs_analysis": 70, 
 "objection_handling": 55, "closing": 45, "mood": "neutral", 
 "mood_reason": "客户对你的数据引用有疑虑"}
-->
```

**前端解析**：

```javascript
function parseScoresFromText(text) {
    // 正则提取 HTML 注释中的 JSON
    var m = text.match(/<!--SCORE\s*(\{[\s\S]*?\})\s*(?:SCORE_END)?\s*-->/);
    if (m) {
        var d = JSON.parse(m[1]);  // 解析 JSON
        // d.professionalism → 65
        // d.mood → "neutral"
        
        // 存入评分历史
        state.scoreHistory.push({
            round: state.msgCount,
            dims: {professionalism: 65, communication: 50, ...}
        });
        
        // 更新右侧评分面板 + ECharts 迷你趋势图
        updateScorePanel(scores);
        updateScoreTrendChart();
    }
}
```

### 为什么用隐藏注释而不是单独 API 调用

> "如果在对话完成后额外调一次评分 API，会增加 50% 的网络请求延迟。把评分数据嵌入回复文本中，通过 HTML 注释传输，前端静默解析——一次请求完成对话+评分，延迟几乎为零。"

## 6. 语音识别（讯飞 WebSocket）

### 是什么

自建 WebSocket 客户端直连讯飞实时语音转写 API，不依赖任何第三方 WS 库。

### 怎么做的

**鉴权（HMAC-SHA256）：**

```python
# 1. 构建签名原文
origin = "host: iat-api.xfyun.cn\n"
origin += "date: Mon, 23 Jun 2026 09:00:00 GMT\n"
origin += "GET /v2/iat HTTP/1.1"

# 2. HMAC-SHA256 签名
signature = base64(hmac_sha256(API_SECRET, origin))

# 3. 拼接 authorization
auth = f'api_key="{API_KEY}", algorithm="hmac-sha256", '
auth += f'headers="host date request-line", signature="{signature}"'

# 4. Base64 + URL 编码后拼入 WebSocket URL
url = f"/v2/iat?authorization={urlencode(base64(auth))}&date={date}&host={host}"
```

**音频流（二进制模式）：**

```
Frame 1 (text):  {"common":{...}, "business":{...}, "data":{"status":0,"audio":""}}
Frame 2~N (binary): 1280 字节 PCM 数据
Last Frame (text):  {"data":{"status":2,"audio":""}}
```

**前端采集**：

```javascript
// AudioContext 16kHz 单声道采集
ctx = new AudioContext({sampleRate: 16000});
processor = ctx.createScriptProcessor(4096, 1, 1);

processor.onaudioprocess = function(e) {
    var float32 = e.inputBuffer.getChannelData(0);
    
    // 降采样（如果实际采样率 ≠ 16000）
    if (actualRate !== 16000) {
        float32 = downsample(float32, actualRate, 16000);
    }
    
    // Float32 → Int16 PCM
    var pcm = new Int16Array(float32.length);
    for (var i = 0; i < float32.length; i++) {
        pcm[i] = float32[i] < 0 ? float32[i] * 0x8000 : float32[i] * 0x7FFF;
    }
    buffer.push(pcm);
};

// 停止后 → 合并所有 PCM → Base64 → POST /api/xfyun-asr
```

### WebSocket 掩码（为什么自己写 WS 客户端）

> "标准 WebSocket 库（如 websocket-client）依赖 C 扩展，会增加部署复杂度。我用原生 socket+ssl 手动实现了 WebSocket 协议的核心部分——握手、分帧、掩码 XOR、收发管理，总共 60 行代码。客户端→服务器必须对 payload 做 4 字节 XOR 掩码，这是协议规定，不加服务器会直接断开。"

## 7. 语音合成（edge-tts）

### 是什么

让 AI 客户用真实人声朗读回复，支持 8 种中文声音。

### 怎么做的

```python
@app.route('/api/tts', methods=['POST'])
def tts():
    text = request.json.get('text', '')
    voice = request.json.get('voice', 'zh-CN-YunxiNeural')
    
    # 调用 edge-tts（微软免费 TTS）
    import edge_tts
    communicate = edge_tts.Communicate(text, voice)
    
    # 流式返回 MP3
    def generate():
        for chunk in communicate.stream():
            if chunk["type"] == "audio":
                yield chunk["data"]
    
    return Response(generate(), mimetype='audio/mpeg')
```

### 为什么用 edge-tts

> "微软的神经网络 TTS，免费，不需要 API Key，中文自然度极高。8 种声音覆盖男女老少各种风格。"

## 8. 知识卡标准化

### 是什么

把 100MB+ 原始文档（Word/PPT/PDF/Excel）拆解为结构化、原子化、可独立检索的知识卡片。

### 怎么做的

**原始文档解析 → 结构化卡片：**

```python
每张知识卡 JSON:
{
    "type": "product",        # 5 种类型之一
    "id": "PROD-001",         # 唯一编号
    "title": "GPU 虚拟化",
    "content": "...",          # 50-200 字，自包含
    "keywords": ["GPU", "虚拟化"],
    "source": "产品手册-云电脑.docx"
}
```

**5 种卡片类型**：

| 类型 | 内容 | 用途 |
|------|------|------|
| product | 产品特性、功能点、参数 | 回答问题"你们能做什么" |
| case | 成功案例（客户名、场景、效果） | 回答"有没有人用过" |
| objection | 常见异议 + 标准回应 | 应对"你们太贵了" |
| solution | 方案要点、适用场景 | 回答"怎么落地" |
| parameter | 技术参数、性能指标 | 回答"具体指标" |

**自动生成流程**：

```python
# batch_generate_cards.py
for scene_dir in knowcard目录:
    for 原始文档 in scene_dir:
        parse_document(原始文档)      # python-docx/PyPDF2 提取文本
        → chunk_by_paragraphs(文本)   # 按段落分块
        → classify_chunks(块)         # LLM 分类为 5 种卡片类型
        → generate_card(块)           # 生成结构化 JSON
    
    # 合并为 knowledge_cards.json
    merge_cards(scene_dir)
```

### 为什么拆卡而不是直接搜文档

> "原始文档一章 5000 字，直接扔给 LLM 有两大问题：一是超过上下文窗口限制；二是包含大量噪音（目录、页眉页脚）。拆成 50-200 字的原子卡后，RAG 可以精准检索到独立的知识点，减少 80% 的无效内容。"

## 11. 对话历史持久化

### 是什么

每次对练结束自动保存到浏览器 localStorage，支持回看、删除。

### 怎么做的

```javascript
var HISTORY_KEY = 'voice_practice_history_v3';

// 保存
function addHistoryRecord(record) {
    var h = loadHistory();                           // 读取已有记录
    h.unshift({                                      // 插入到最前面
        id: Date.now(),
        time: new Date().toISOString(),
        scene: state.selectedScene,
        rounds: state.msgCount,
        score: {avg: 75, dims: {...}},               // 保存评分
        messages: state.messages.slice(-20)           // 保留最后 20 轮
    });
    if (h.length > 50) h = h.slice(0, 50);           // 最多 50 条
    localStorage.setItem(HISTORY_KEY, JSON.stringify(h));
}
```

**事件委托方式绑定点击（避免 IIFE 作用域问题）：**

// ❌ 错误：onclick 在全局作用域找不到 IIFE 内的函数

## 常见追问 & 应答

### Q: 大模型调一次多少钱？

> "DeepSeek-V3 通过 SiliconFlow 调用，大约 0.001 元/千 token。单轮对话 300-400 token，成本约 0.0004 元。一场 10 轮的对练，模型成本不到 1 分钱。"

### Q: 并发能撑多少？

> "Flask 单进程 + gunicorn 4 worker，轻量级 API 调用可以撑 50-100 并发。实际瓶颈在 DeepSeek API 限速，不在我们服务端。"

### Q: 你的 RAG 和向量检索差多少？

> "语义理解上向量检索更好，但我们的场景有特殊性——知识卡高度结构化，产品名和参数名是精确关键词。在这种场景下，关键词+短语匹配的准确率已经达到实用水平。如果需要升级，方案是 sentence-transformers + ChromaDB，工作量约 3 天。"

### Q: 如果知识库更新了，需要重启服务吗？

> "不需要。每次对话时实时调用 `rag.load_scene()` 加载最新文档，新增文档即时生效。"

### Q: 安全方面考虑了哪些？

> "API Key 存储在 `.env` 文件，通过环境变量读取；前端不做任何鉴权（Demo 阶段）；生产部署会加 HTTPS + Token 认证。"

---

> **答辩口诀**：每个技术点 → 先说"做了什么" → 再说"为什么这么做（tradeoff）" → 最后留一句"下一步怎么优化"。不说空话，不回避技术债，展示工程判断力。
