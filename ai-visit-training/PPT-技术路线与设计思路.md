## 1. 项目背景与痛点

### 售前人员的真实困境

| 痛点 | 现状 | 后果 |
|------|------|------|
| 🎯 **缺乏实战训练** | 新人只能"跟访"老销售，被动学习 | 成长周期 6-12 个月 |
| 📚 **知识孤岛** | 产品资料分散在 50+ 个 Word/PPT/Excel | 临场找不到依据 |
| ❌ **试错成本高** | 真实客户面前犯错不可逆 | 丢单率 30%+ |
| ⏰ **时间不灵活** | 角色扮演需要两人同时在线 | 练习频次低 |
| 📊 **无法量化** | 没有客观的能力评估体系 | 不知道自己的短板 |

### 行业现状

- B2B 销售拜访是**高度经验驱动**的工作
- 每个客户拜访周期（初次接触→需求深挖→方案呈现→异议谈判→促成收尾）需要的核心能力完全不同
- 传统培训无法模拟真实客户的多变反应和刁难

## 5. 五阶段对练体系

### 设计理念

B2B 销售拜访天然存在 5 个递进阶段，每个阶段客户心理状态不同，对销售人员的能力要求也不同。传统对练无法区分阶段，导致训练没有针对性。

### 阶段架构

```
客户拜访完整周期
│
├─ 阶段1: 初次接触 👋 ──── 3-5 轮
│   ├ 客户状态: 礼貌但戒备
│   ├ 评分侧重: 表达沟通 40% + 需求洞察 30%
│   ├ AI 规则: 回复简短、不主动展开、不透露内部信息
│   └ 话术模板: "你们主要做哪块的？""大概说说吧"
│
├─ 阶段2: 需求深挖 🔍 ──── 4-6 轮
│   ├ 客户状态: 开始敞开心扉但有所保留
│   ├ 评分侧重: 需求洞察 50% + 沟通 25%
│   ├ AI 规则: 分享部分情况、核心痛点闪烁其词
│   └ 话术模板: "确实有这个问题...但具体影响说不准"
│
├─ 阶段3: 方案呈现 📊 ──── 5-6 轮
│   ├ 客户状态: 期待方案但对效果存疑
│   ├ 评分侧重: 专业度 40% + 需求洞察 25%
│   ├ AI 规则: 追问每个模糊说法、反感 PPT 大词
│   └ 话术模板: "有实际案例吗？""别讲概念，说实际的"
│
├─ 阶段4: 异议谈判 ⚖️ ──── 5-8 轮
│   ├ 客户状态: 强势挑剔，频繁提竞品压价
│   ├ 评分侧重: 异议处理 50% + 专业度 25%
│   ├ AI 规则: 每轮必提竞品/价格/风险
│   └ 话术模板: "B公司比你们低30%，凭什么？"
│
└─ 阶段5: 促成收尾 🤝 ──── 3-5 轮
    ├ 客户状态: 接近决策但需临门一脚
    ├ 评分侧重: 成交引导 50% + 沟通 25%
    ├ AI 规则: 纠结落地风险和 ROI
    └ 话术模板: "出了问题多久解决？""从签合同到落地要多久？"
```

### 差异化配置

每个阶段都有 **5 条专属行为规则 + 4 条话术模板 + 背景故事 (pain_points)**，深度注入到 System Prompt，确保 AI 客户行为与阶段精准匹配。

### 示例：异议谈判阶段完整配置

```
背景故事:
"采购部老张跟XX公司合作5年了铁得很，你想推新供应商但他不买账；
另外你们是国企，超50万必须公开招标，流程至少3个月；
如果销售只报价格不给绕过招标的合规方案，你就没法往下聊了"

专属规则:
1. 每轮必须提到竞品名、价格差异或实施风险
2. 口头禅："XX公司比你们...，你们优势在哪？"
3. 对模糊的优势陈述立刻打回——"你说的这些XX也能做到"
4. 测试压力下的专业度和情绪控制
5. 给一个优势然后质疑另一个，不完全接受

话术模板:
- "B公司报价比你们低30%，用的也是微服务架构，凭什么？"
- "你说的智能运维XX也有啊，还比你们多一个功能"
- "XX跟我们合作三年了，换的成本你考虑过吗？"
- "你们有什么是别人做不到的？说一个就行"
```

### 反转模式（一键切换）

```
┌──────────────────────────────────────────┐
│  用户扮演客户 ←→ AI 扮演资深售前经理        │
│                                          │
│  ○ 用户随意向 AI 提问                      │
```

- 页面状态栏右侧大按钮 **「🔄 反转模式」** 一键切换
- 反转模式 → 左右栏隐藏，对话区全屏
- 场景/阶段选择透明化，纯自然对话

## 7. RAG 知识库检索

### 为什么自研 SimpleRAG？

| 方案 | 优点 | 缺点 |
|------|------|------|
| 向量数据库 (ChromaDB) | 语义召回好 | 需要额外服务、GPU 嵌入 |
| ElasticSearch | 全文检索强 | Java 依赖、部署重 |
| **SimpleRAG** | **零依赖、部署轻、中文友好** | 语义理解弱于向量检索 |

### 三维度混合打分算法

```
相关性分数 = 关键词命中 × 50% + 短语匹配 × 35% + Jaccard × 15%

① 关键词命中 (50%)
  - 精确匹配 (权重 1.0): 完整词命中
  - 部分匹配 (权重 0.5): 关键词包含于内容
  - 全文出现 (权重 0.2): 在整个 chunk 中出现

② 短语匹配 (35%)
  - 2-4 字连续滑动窗口匹配
  - 越长短语权重越高 (length × 1.5)

③ Jaccard 相似度 (15%)
  - 字符集交集 / 并集
  - 兜底策略，确保基础召回
```

### 知识库组织结构

```
knowcard/                    ← 主要知识库 (50+ 场景)
├── AI/                      ← 按产品大类分组
│   ├── 合同审查大模型/        ← 单个场景
│   │   ├── scene_card.json   ← 场景卡 (人设/痛点/产品)
│   │   ├── knowledge_cards.json ← 知识卡 (产品/案例/异议等)
│   │   ├── 产品介绍.docx
│   │   └── 竞品分析.pdf
│   └── 数据治理服务/
├── 标品/
│   ├── 云电脑/
│   └── 安全大脑/
└── 解决方案/
    ├── 数字工厂-智慧仓储/
    └── IT运维服务方案/

每个场景包含:
├── scene_card.json     → 场景元信息
├── knowledge_cards.json → 5类结构化卡片
│   ├── product    ← 产品特性
│   ├── case       ← 成功案例
│   ├── objection  ← 常见异议及回应
│   ├── solution   ← 方案要点
│   └── parameter  ← 技术参数
└── 原始文档           → .docx/.pdf/.pptx/.xlsx/.txt
```

### 检索策略

```
用户提问 → 中文分词 → 提取关键词
  ↓
优先匹配场景名（如"云电脑"→ 定位到标品/云电脑/）
  ↓
加载该场景 knowledge_cards.json + 解析原始文档
  ↓
三维度打分 → 排序 → 去重 → top_k 截断
  ↓
注入 System Prompt 作为上下文参考
```

## 8. 实时评分与可视化

### 五维评分体系

| 维度 | ID | 含义 | 满分 |
|------|-----|------|:--:|
| 🎯 专业度 | professionalism | 产品/技术知识的准确性和深度 | 100 |
| 💬 表达沟通 | communication | 语言清晰度、逻辑性、共情能力 | 100 |
| 🔍 需求洞察 | needs_analysis | 挖掘隐性需求、精准提问的能力 | 100 |
| 🛡️ 异议处理 | objection_handling | 化解质疑、竞品对比的应对能力 | 100 |
| ✅ 成交引导 | closing | 推动决策、明确下一步的能力 | 100 |

### 评分传输机制

AI 回复中嵌入 HTML 注释隐藏标记：

```
AI 回复内容:
"你说得对，我们确实在考虑成本问题..."

<!--COACH
本轮你直接驳回了客户的价格质疑但没有先共情，
建议先说'您的预算考虑非常合理'再切入差异化价值。
-->

<!--SCORE
{
  "professionalism": 75,
  "communication": 60,
  "needs_analysis": 70,
  "objection_handling": 55,
  "closing": 50,
  "mood": "neutral",
  "mood_reason": "客户对你的数据引用有疑虑"
}
-->
```

前端通过 `parseScoresFromText()` 正则提取，**评分数据对用户不可见**。

### ECharts 可视化

1. **右侧栏迷你趋势图**：五维评分折线，对练中实时更新
2. **最终报告雷达图**：五维能力画像，直观展示强弱项
3. **最终报告趋势图**：全轮次评分走势，自动标注 🏆 最佳轮次 / ⚠️ 最差轮次

### 客户态度追踪

每轮 AI 会评估客户当前情绪：

| 情绪 | 图标 | 含义 |
|------|:--:|------|
| 😊 happy | 绿 | 客户满意感兴趣 |
| 😐 neutral | 黄 | 犹豫考虑中 |
| 😡 angry | 红 | 有顾虑不满 |

## 9. 语音交互闭环

### 语音输入

```
用户说话 → 浏览器 Web Speech API
  → interimResults (实时预览)
  → final transcript (确认结果)
  → 自动填充输入框并发送
```

- `continuous: true` — 持续识别
- `interimResults: true` — 实时临时结果
- 支持 Chrome/Edge 原生语音识别

### 语音输出

```
AI 回复文本 → edge-tts (微软神经网络 TTS)
  → MP3 流 → Web Audio API 播放
```

可选声音（8 种）：

### 两种部署模式

```
模式 A: 轻量模式 (默认)
  ├─ 不加载 FunASR (省 2GB 内存)
  ├─ 前端 Web Speech API 识别
  └─ 适合：快速部署、低成本运行

模式 B: 完整模式 (--full)
  ├─ 加载 FunASR 语音识别
  ├─ 服务端处理 Base64 音频
  └─ 适合：Firefox/Safari 等不支持 Web Speech API 的浏览器
```

## 11. 创新点总结

| # | 创新点 | 说明 |
|---|--------|------|
| 1 | **五阶段渐进式对练** | 将客户拜访拆解为 5 个独立阶段，每阶段有独立的人设、规则、评分侧重 |
| 2 | **双模式角色切换** | 常规模式 (训练) + 反转模式 (学习)，一正一反双向覆盖 |
| 3 | **隐藏标记评分** | `<!--COACH--><!--SCORE-->` 实现评分数据与对话内容分离传输 |
| 4 | **轻量级 RAG** | 纯 Python 三维度混合检索，零外部依赖，部署极简 |
| 5 | **开挂模式** | 双路 RAG 检索增强召回，实时提供精准销售话术参考 |
| 6 | **知识卡标准化** | 100MB+ 文档拆解为 5 类原子化卡片，可独立检索 |
| 7 | **ECharts 多维度可视化** | 雷达图 + 趋势图 + 最佳/最差标注，能力量化一目了然 |
| 8 | **localStorage 历史回看** | 完整保存 + 一键回看任意历史对练，形成个人成长轨迹 |
| 9 | **Apple HIG  UI 设计** | 遵循 Apple Human Interface Guidelines 卡片风格，清爽专业 |
| 10 | **语音交互闭环** | 语音输入 (Web Speech API) + 语音输出 (edge-tts)，全语音对练 |
