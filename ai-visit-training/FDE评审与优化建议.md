## 一、建议优化的方向（按优先级排列）

### 🥇 P0 - 门面工程（答辩加分项）

#### 1.1 代码模块化拆分

**现状**：`app.py` 1500+行，所有逻辑混在一个文件，导师和评委很难快速理解结构。

**优化**：拆分为模块化的包结构
```
demo2/
├── app.py                    # 入口 (50行)
├── config.py                 # 配置 (模型/路径/密钥)
├── rag/
│   ├── __init__.py
│   ├── retriever.py          # SimpleRAG类
│   └── document_parser.py    # _extract_text()
├── api/
│   ├── __init__.py
│   ├── chat.py               # /api/chat
│   ├── cheat.py              # /api/cheat
│   ├── scenes.py             # /api/scenes
│   └── tts.py                # /api/tts
├── prompts/
│   ├── customer.py           # 客户角色Prompt
│   └── coach.py              # 教练/开挂Prompt
├── static/
│   └── index.html
└── tests/
    └── test_rag.py
```

**FDE加分理由**：展示软件工程素养，"我能写出可维护的生产级代码"。

#### 1.2 README.md 重写（面向答辩）

### 🥈 P1 - 功能完善（展示思考深度）

#### 2.1 对话历史管理

**现状**：刷新页面对话丢失，无历史记录。

**优化**：
- 对话结束后保存到浏览器 localStorage
- 新增"历史记录"列表，可回看评分变化曲线
- 单次对话结束后，用 ECharts 画五维雷达图

**FDE加分理由**：体现数据驱动思维，"不只是对练，还能跟踪进步"。

#### 2.2 语音体验优化

**现状**：TTS自动播报但无法中断，语音识别有时不准确。

**优化**：
- 增加"暂停播报"按钮
- 语音输入增加实时转写校对（允许手动修正再发送）
- 完整模式下 FunASR 热词配置（场景相关产品名作为热词）

#### 2.3 错误处理增强

**现状**：API失败只显示"请求失败"，没有重试提示。

**优化**：
- 网络断开时显示 Toast 提示"网络连接中断，正在重试..."
- API Key 失效时友好提示"请检查API Key配置"
- 增加 session 超时自动保存

### 🥉 P2 - 技术深度（体现专业能力）

#### 3.1 RAG检索升级

**现状**：简单的关键词匹配 + Jaccard 相似度。

**优化方向（可选1个展示）**：
- **方案A**：接入 `sentence-transformers` 做向量语义检索（准确率提升30%+）
- **方案B**：HyDE（Hypothetical Document Embeddings）——先让LLM生成假设答案，再用假设答案去检索
- **方案C**：BM25 + 向量检索混合（既有关键词又有语义）

**FDE加分理由**：展示对大模型生态的理解深度。

#### 3.2 多轮对话记忆优化

**现状**：简单截断最近12条，会丢失早期重要信息。

**优化**：使用 LLM 自动做对话摘要
```
每6轮对话 → 调LLM生成50字摘要 → 将摘要作为系统消息注入
```
这样既能保持长对话连贯性，又不会超出 token 限制。

#### 3.3 评分体系可视化

**现状**：右侧栏简单显示分数条。

**优化**：
- 每轮对话后的评分变化做成折线图（X轴=轮次，Y轴=分数）
- 结束报告增加 ECharts 五维雷达图
- 标注"最佳轮次"和"最差轮次"
