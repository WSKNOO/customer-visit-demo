央国企客户拜访情报助手 Demo
需求说明与技术路线建议（学员版）
一句话目标
• 产品经理输入央国企客户名称后，系统自动联网检索公开信息，生成客户画像、近期动态、数字化线索、潜在需求、产品匹配点和拜访建议。
• 本阶段只要求完成 Demo：能跑通流程、输出结构化简报、带来源链接；不追求正式商用级稳定性。

1. 项目背景与需求理解
在面向央国企客户开展产品交流、方案推荐和商机挖掘前，产品经理通常需要提前了解客户的企业背景、主营业务、近期动态、数字化建设方向、招投标线索和可能痛点。
这些信息分散在客户官网、国资委网站、政府新闻、招投标平台、公众号、媒体报道等公开渠道，人工检索耗时较长，也容易遗漏关键信息。因此希望通过“大模型联网检索 + 结构化分析”的方式，辅助产品经理快速完成拜访前准备。
2. Demo 目标范围
本 Demo 不要求一次性做成正式系统，重点是让学员理解完整业务流程，并做出一个可演示、可复用的最小版本。
输入：客户名称、拜访目的、关注方向。
处理：联网搜索公开信息，抓取正文，筛选与客户拜访相关的内容。
分析：让大模型按固定模板总结客户情况，判断数字化/智能化线索。
输出：生成一份结构化客户拜访简报，包含来源链接和不确定性提示。
边界：Demo 阶段只使用公开互联网信息；内部产品库可先用一个简化版产品能力表模拟。

5. 推荐技术路线：先做 Demo，不做重系统
推荐采用“搜索 API + 网页抓取清洗 + 大模型分析 + 模板化输出”的轻量路线。不要一开始就做复杂知识图谱、复杂权限系统或全量企业数据库。
环节	推荐做法	说明
前端输入	简单 Web 页面、Dify 表单或命令行均可	先保证流程跑通，不纠结界面美观
联网搜索	搜索 API 或定向爬取	搜索客户名称 + 数字化/招标/AI/云/安全等关键词
网页抓取	requests / BeautifulSoup / Playwright	读取网页正文，过滤导航、广告、重复内容
内容清洗	按来源、时间、关键词进行筛选	优先保留官网、政府、国资委、招投标、权威媒体
大模型分析	固定 Prompt + 结构化输出	要求事实带来源，推测单独标注
产品匹配	简化产品能力表或内部知识库检索	Demo 阶段可用 Excel/JSON 模拟
结果输出	Markdown/HTML/Word	建议先输出 Markdown，再扩展 Word 导出

6. Demo 架构流程
建议学员按下面的流程理解和实现，不需要一开始追求复杂 Agent，只要每一步职责清楚即可。
流程图（文字版）
• 用户输入客户名称、拜访目的、关注方向。
• 系统生成多组搜索关键词，例如“客户名称 + 数字化转型”“客户名称 + 招标”“客户名称 + 人工智能”。
• 调用搜索接口，获得候选网页标题、摘要、链接和发布时间。
• 抓取候选网页正文，并进行清洗、去重、过滤。
• 将有效材料交给大模型，抽取客户画像、近期动态、数字化线索。
• 检索或读取简化产品能力表，匹配可切入产品。
• 大模型生成结构化客户拜访简报，并附来源链接和不确定性说明。

7. 大模型联网的关键要求
要求	具体说明	Demo 实现建议
搜索可控	不能只让模型自由搜索，要设计关键词和来源优先级。	预置搜索 query 模板
来源可信	央国企场景优先使用官网、政府、国资委、招投标和权威媒体。	为来源设置简单权重
时间有效	客户动态要关注近期，基础信息和领导信息要注意是否过期。	默认检索近一年，重要规划可放宽到三年
引用溯源	关键事实必须能看到来自哪个网页。	输出中保留链接和标题
事实与推测分开	不能把“可能需求”写成“明确需求”。	输出中单设“推测判断”模块
客户名称消歧	简称、重名企业容易搜错。	必要时列出候选企业，让用户确认
低置信度提示	查不到或证据不足时要明确说明。	Prompt 中要求“不确定就说不确定”

9. 推荐输出模板
为了避免大模型自由发挥，建议固定输出结构。Demo 输出可以采用如下模板：
输出模块	内容要求
一、客户基本画像	企业性质、所属行业、主营业务、上级集团、区域布局等。
二、近期重点动态	按时间倒序列出近期新闻、公告、签约、项目、会议等。
三、数字化/智能化线索	提取与 AI、大数据、云、算力、安全、5G、物联网、信创相关的信息。
四、潜在业务痛点	基于行业特点和公开信息归纳可能痛点。
五、可能匹配的产品能力	结合内部产品能力表，列出可推荐方向。
六、拜访切入点	给出适合开场的话题和沟通切入方式。
七、建议提问清单	列出拜访时可以向客户确认的问题。
八、后续商机判断	判断可跟进方向、优先级和注意事项。
九、风险与不确定性	说明哪些内容证据不足、需要人工确认。
十、信息来源	列出标题、链接、发布时间、来源类型。

10. Prompt 设计建议
Demo 阶段可以用固定 Prompt 约束模型，重点强调“只基于检索材料回答”“事实和推测分开”“关键结论带来源”。
可参考 Prompt
• 你是一名政企客户售前分析助手，请基于给定的联网检索材料，为产品经理生成客户拜访前简报。
• 只允许使用材料中出现的信息作为事实，不得编造。
• 对于潜在需求、产品匹配、商机判断，请明确标注为“基于公开信息的推测”。
• 重要事实后需要标注来源标题或链接编号。
• 如果材料不足，请写明“未检索到明确证据”，不要强行下结论。
• 输出结构必须包含：客户画像、近期动态、数字化线索、潜在需求、产品匹配、拜访建议、提问清单、风险提示、信息来源。

11. 学员实现任务拆分
任务	负责人能力要求	验收标准
搜索模块	会调用 API 或写简单爬虫	输入客户名称后能返回 10-20 条相关网页
网页解析模块	会处理 HTML/PDF 基础解析	能提取正文、标题、链接、发布时间
信息筛选模块	了解关键词和来源权重	能过滤明显无关、过期、低可信内容
大模型分析模块	会写 Prompt 和调用模型 API	能输出结构化客户简报
产品匹配模块	理解产品能力与需求映射	能用静态产品表匹配 3-5 个建议方向
结果展示模块	会做简单页面或 Markdown 输出	结果可读、结构清楚、来源可点击

12. Demo 验收标准
建议选择 1-2 家典型央国企客户进行演示，每家客户能生成一份简报。验收时不追求内容绝对完整，但要看流程是否完整、结论是否有依据。
1.能输入客户名称并自动触发联网检索。
2.能展示至少 5 条有效来源链接。
3.能生成结构化客户拜访简报。
4.能识别至少 3 条与数字化、智能化或信息化相关的线索。
5.能给出 3-5 个可能的产品切入方向。
6.能生成拜访开场话题和建议提问清单。
7.事实信息和推测判断有明显区分。
8.查不到信息时不会胡编。
13. 注意事项
本 Demo 仅使用公开互联网信息，不接触客户内部敏感数据。
模型输出不能直接作为正式商务材料，重要内容需要人工复核。
不要让模型直接生成“客户一定需要购买某产品”的武断结论。
对领导信息、组织架构、招投标信息等容易变化的内容，要标注发布时间。
若客户名称存在简称或重名，系统应提示用户确认目标企业。
正式上线前还需要补充权限、日志、合规、安全和内网部署方案。
14. 最终建议
建议本阶段目标定位为“客户拜访前情报准备 Demo”，而不是完整商用系统。先跑通客户名称输入、联网搜索、资料清洗、大模型分析、产品匹配和简报输出这条主链路。
如果 Demo 效果得到认可，下一步再考虑接入内部产品库、方案库、成功案例库，并增加 Word/PPT 导出、用户权限、日志审计、缓存和人工校验流程。
